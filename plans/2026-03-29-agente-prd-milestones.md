# agentE 产品需求与里程碑文档 v0

## 1. 项目背景

agentE 的目标是把用户的一段自然语言输入，转成可渲染、可预览、可导出的视频内容。当前你给出的目标链路是：

`自然语言 -> TSX 模板代码 / 网页动画 -> headless 多浏览器截图 -> 60fps 视频`

结合当前仓库，已确认的事实如下：

- 当前仓库已经有基于 Remotion 的静态 composition 能力，核心入口在 `src/Root.tsx`。
- 当前仓库已经有一条可执行的渲染脚本链路，见 `scripts/render-showcase-video.ps1`，说明 `Remotion -> 视频导出` 这一段已经有基础能力。
- 当前仓库暂未看到 Next.js 应用、GPT API 编排、模板编译器、任务队列、浏览器截图调度层。
- 你期望使用 `pnpm` 管理依赖，但当前仓库仍以 `package-lock.json` 和 `npm` 命令为主，这需要在首个里程碑统一。

因此，agentE 的核心问题不是“能不能渲染视频”，而是“如何把自然语言稳定地转成可控、可重复、可审计的 TSX/动画产物”。

## 2. 产品目标

### 2.1 MVP 目标

MVP 阶段聚焦以下结果：

- 用户输入一句自然语言，系统能生成一个可预览的视频草稿。
- 草稿由受控的 TypeScript/TSX 模板系统生成，而不是完全自由的任意代码执行。
- 用户能看到生成结果、基础元信息和失败原因。
- 最终可以导出 1080p、60fps 的视频文件。

### 2.2 非目标

以下能力不建议放入第一阶段：

- 无约束地生成任意 React/TSX 代码并直接执行。
- 支持过于开放的“任意网站录屏”能力。
- 复杂多角色配音、自动 BGM 编排、AI 作曲。
- 完整在线协作编辑器和团队工作流。

## 3. 推荐产品路线

### 3.1 三种可选方案

**方案 A：LLM 直接输出完整 TSX 代码**

- 优点：实现快，演示效果直接。
- 缺点：结果不稳定，容易生成不可编译代码；样式、动画、资源引用也很难约束；线上执行风险高。

**方案 B：LLM 先输出结构化 `VideoSpec`，再由模板编译器生成 TSX**

- 优点：最适合产品化；便于校验、审计、回放和迭代；可以逐步扩展模板能力。
- 缺点：前期要先设计 DSL/Schema 和编译器。

**方案 C：只做模板参数填充，不允许生成新结构**

- 优点：最稳，最容易上线。
- 缺点：创造力不足，难以承接复杂 prompt。

### 3.2 推荐结论

建议采用 **方案 B**，并在第一阶段保留少量方案 C 的能力作为兜底。

即：

- LLM 不直接生成最终 TSX 文件。
- LLM 先产出结构化 `VideoSpec`。
- 平台根据 `VideoSpec` 选择模板、拼接 scene、注入 props、编译出 TSX。
- 对无法稳定映射的复杂需求，降级为“推荐模板 + 待补充信息”。

这样可以在“创意自由度”和“工程可控性”之间取得平衡。

## 4. 目标用户与核心场景

当前文档先按以下假设定义目标用户：

- 需要快速生成产品介绍视频的个人开发者或小团队。
- 需要生成功能演示、宣传短片、产品更新视频的运营/增长团队。
- 需要把一段产品文案快速变成动态视频草稿的内部内容团队。

MVP 优先覆盖的场景：

- SaaS 产品介绍视频
- 新功能发布视频
- App/网站功能演示视频
- 简单品牌口播风格的动态短视频

不建议第一阶段覆盖的场景：

- 电影级叙事视频
- 高自由度 3D 动画
- 长时长、多镜头、强剪辑的视频工程

## 5. 核心用户流程

### 5.1 端到端流程

1. 用户输入自然语言 prompt。
2. 系统调用 GPT API，将 prompt 转成结构化 `VideoSpec`。
3. 系统选择合适模板或 scene 组合。
4. 模板编译器根据 `VideoSpec` 生成受控的 TypeScript/TSX 代码与 props。
5. 系统启动预览渲染，生成网页动画预览。
6. 对需要浏览器真实渲染的部分，使用 headless 浏览器进行截图或帧捕获。
7. 将捕获素材与 Remotion composition 合成，交给 ffmpeg 输出最终视频。
8. 返回预览链接、导出文件和日志信息。

### 5.2 建议的数据流

建议把内部数据流拆成 4 层：

- `PromptInput`：用户原始输入
- `VideoSpec`：LLM 生成的结构化视频意图
- `RenderPlan`：平台归一化后的可执行渲染计划
- `RenderArtifact`：预览、日志、帧图、最终视频等产物

这样做的好处是每一层都能缓存、调试和回放。

## 6. 功能需求

### 6.1 Prompt 理解层

必须具备：

- 从自然语言中提取视频目标、风格、时长、比例、语言、核心文案。
- 支持缺失信息自动补全，但要明确标记哪些字段是系统推断。
- 当 prompt 过于模糊时，返回结构化追问，而不是直接胡乱生成。

建议输出的 `VideoSpec` 至少包含：

- `projectType`
- `durationSec`
- `aspectRatio`
- `language`
- `brandTone`
- `scenes[]`
- `assets[]`
- `cta`

### 6.2 模板系统

必须具备：

- 模板注册机制
- 场景级 scene 复用机制
- 模板参数 Schema 校验
- 模板版本号和回滚能力

建议第一阶段模板类型：

- Hero 开场
- Feature Highlight
- KPI / 数据卡片
- Timeline / Step Flow
- CTA 收尾

### 6.3 TSX 代码生成层

必须具备：

- 从 `VideoSpec` 到 TSX 的确定性编译逻辑
- TypeScript 类型检查
- 模板白名单
- 样式和动画 API 白名单

不建议第一阶段支持：

- 任意第三方 npm 包动态注入
- 任意网络资源直接拉取执行
- LLM 输出未经审核的完整业务逻辑

### 6.4 预览与导出

必须具备：

- 单次生成任务的状态机：`queued -> generating -> validating -> previewing -> rendering -> completed/failed`
- 预览图或低成本预览视频
- 最终视频导出
- 基础错误日志和失败原因回传

### 6.5 Headless 浏览器捕获

建议只在以下场景使用真实浏览器捕获：

- DOM/CSS 动画依赖浏览器布局结果
- 需要网页真实渲染效果
- 需要录制 canvas 或复杂前端动画

如果只是标准模板动画，优先直接用 Remotion 内部渲染，避免无意义的浏览器录屏成本。

## 7. 非功能需求

### 7.1 性能

- 目标输出：1080p / 60fps
- MVP 目标时延：普通 15-30 秒视频在可接受时间内完成导出
- 同一模板重复渲染应尽量复用 bundle 或缓存中间产物

### 7.2 稳定性

- 同一 `VideoSpec` 多次执行结果应基本一致
- 字体、资源、动画时间轴要可复现
- 任务失败要可重试、可定位

### 7.3 安全

- 禁止直接执行 LLM 输出的任意代码
- 模板运行环境与 API 服务隔离
- 资源访问需要白名单或代理层

### 7.4 可观测性

- 每次生成保留 prompt、spec、模板版本、渲染日志、错误快照
- 能区分 LLM 失败、编译失败、浏览器捕获失败、视频导出失败

## 8. 建议技术架构

### 8.1 服务拆分

建议按以下模块拆分：

- `web`：Next.js 前端，负责输入、任务列表、预览、导出
- `api`：Next.js Route Handlers 或独立 API 服务，负责调用 GPT API、任务提交、状态查询
- `template-engine`：TypeScript 包，负责 `VideoSpec -> TSX/props`
- `renderer`：Remotion/ffmpeg 渲染服务
- `capture-worker`：headless 浏览器截图或帧捕获服务

### 8.2 推荐目录方向

如果继续在当前仓库演进，建议逐步调整为：

```text
apps/
  web/                 # Next.js
packages/
  video-templates/     # Remotion compositions / scenes
  template-engine/     # VideoSpec -> TSX / props
  shared-schema/       # Zod / TypeScript types
workers/
  renderer/
  capture/
```

如果暂时不做 monorepo，也至少要把“模板”和“编译器”从页面层分离出来。

## 9. 里程碑规划

以下里程碑按“先把链路做通，再把自由度放开”的策略设计。

### M0：工程基线统一

目标：

- 明确仓库形态和运行方式，统一到 `pnpm`
- 引入 Next.js 基础壳
- 设计 `VideoSpec` Schema

交付物：

- `pnpm-lock.yaml`
- Next.js 应用骨架
- 统一的环境变量说明
- `VideoSpec` v0 类型定义

验收标准：

- 本地可同时启动 web 与渲染模块
- 可以提交一个固定 mock spec 并完成预览或导出

### M1：自然语言 -> 结构化 Spec

目标：

- 接入 GPT API
- 从 prompt 生成受控 `VideoSpec`
- 增加基础追问/澄清机制

交付物：

- prompt parser
- `VideoSpec` 生成接口
- spec 校验与错误提示

验收标准：

- 对 10 条典型 prompt，至少能稳定生成合法 spec
- 失败时能返回结构化错误而不是崩溃

### M2：Spec -> TSX 模板编译

目标：

- 建立模板注册表
- 完成 `VideoSpec -> TSX/props` 编译器
- 支持至少 3-5 个基础 scene

交付物：

- 模板编译器
- 模板白名单
- TS 类型校验与构建校验

验收标准：

- 合法 spec 都能转成可渲染模板
- 非法字段会在编译前被拦截

### M3：预览与浏览器捕获

目标：

- 生成可回看的预览
- 接入 headless 浏览器进行真实动画捕获
- 建立任务状态与产物管理

交付物：

- 任务状态机
- 预览页
- 浏览器捕获 worker
- 中间帧或中间视频产物管理

验收标准：

- 用户可查看任务进度
- 复杂 DOM 动画可通过浏览器捕获进入最终渲染链路

### M4：Remotion + ffmpeg 生产化集成

目标：

- 把现有 Remotion/ffmpeg 能力接入产品链路
- 输出 60fps 最终视频
- 补足错误恢复和缓存策略

交付物：

- 渲染编排层
- 产物目录/存储规范
- 重试与失败恢复策略

验收标准：

- 从自然语言到最终视频形成完整闭环
- 至少 3 类模板能稳定导出 60fps 视频

### M5：Beta 强化

目标：

- 提升稳定性、模板丰富度和运营可用性

交付物：

- 模板扩展
- 任务历史
- 参数微调
- 基础运营后台或任务面板

验收标准：

- 外部用户可连续使用，不需要工程师手工介入

## 10. 当前最重要的工程决策

以下决策会直接影响后续实现成本：

1. 是否坚持“LLM 先生成 `VideoSpec`，而不是直接生成 TSX”。
2. 第一阶段是否只做模板化视频，而不是开放式视频生成。
3. 浏览器捕获是否仅用于少数复杂动画，而不是所有视频都走浏览器录制。
4. 是否立刻迁移到 Next.js + `pnpm`，还是先在当前 Remotion 仓库内把编译器做出来。

当前建议：

- 选择 `VideoSpec -> 编译器 -> TSX` 的路线
- 先做模板化 MVP
- 保留浏览器捕获为增强能力，不作为默认路径
- 先统一工程基线，再进入生成链路开发

## 11. 风险与依赖

### 11.1 主要风险

- LLM 输出不稳定，导致 spec 质量波动
- 直接生成 TSX 会放大编译失败和安全问题
- 浏览器帧捕获成本高，吞吐不稳定
- 字体、图片、外链素材会影响渲染一致性

### 11.2 外部依赖

- GPT API 调用额度与响应延迟
- 浏览器运行环境
- ffmpeg 可执行环境
- 字体与静态资源管理方案

## 12. 待你确认的关键上下文

以下问题会决定 agentE 的 MVP 边界：

- 第一阶段主要面向哪类视频：产品介绍、功能演示、营销短片，还是通用视频生成？
- 用户是否可以上传素材，还是先只支持纯文本输入？
- 第一阶段是否需要旁白、字幕、BGM？
- 输出是否只做 16:9，还是同时支持 9:16 / 1:1？
- 你希望用户拿到的是“全自动成片”，还是“先出草稿再编辑”？
